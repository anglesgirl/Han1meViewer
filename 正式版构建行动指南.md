# Han1meViewer 正式版构建行动指南

本文面向需要自行构建和发布 Han1meViewer 正式版的维护者。正式版与调试版的关键差异是：**必须使用稳定的发布密钥签名**，并将该密钥对应的签名证书 SHA-256 回填到原生校验逻辑中。否则应用在正式版的视频页会触发签名校验失败。

> 不要使用 Android Studio 自动生成的临时调试密钥发布正式版。发布后的更新必须始终使用同一把发布密钥；丢失密钥后，已安装旧版的用户无法直接安装新版本覆盖升级。

## 1. 构建前准备

### 1.1 所需环境

- Android Studio：使用能识别 Android Gradle Plugin 9.2.1 的版本。
- JDK：项目编译目标为 Java 21，建议在 Android Studio 的 Gradle JDK 中选择 JDK 21。
- Android SDK Platform：Android API 37。
- Android SDK Build-Tools：安装与 API 37 配套的最新版。
- NDK (Side by side)：项目包含 `app/src/main/cpp/` 下的原生代码，必须安装 NDK。
- CMake：安装 3.22.1；项目在 `app/build.gradle.kts` 中固定使用该版本。

可在 Android Studio 的 **Settings > Languages & Frameworks > Android SDK > SDK Tools** 中安装 NDK 和 CMake。首次执行 Gradle 构建还会下载 Gradle Wrapper 与 Maven 依赖，请确保网络可访问 `google()`、`mavenCentral()` 和 JitPack。

### 1.2 当前项目构建参数

以下参数来自 `app/build.gradle.kts`，修改发布版本时应一并检查：

| 项目 | 当前配置 | 说明 |
| --- | --- | --- |
| `applicationId` | `io.github.daisukikaffuchino.han1meviewer` | 正式版包名；不要随意改动，否则不能覆盖安装旧版。 |
| `minSdk` | 29 | 仅支持 Android 10 及以上。 |
| `targetSdk` / `compileSdk` | 37 / 37 | 需安装对应 Android SDK。 |
| Java / Kotlin JVM target | 21 / 21 | Gradle JDK 必须匹配。 |
| ABI | `arm64-v8a` | Release 仅产出 64 位 ARM APK，不支持 `armeabi-v7a`、x86 或 x86_64。 |
| 混淆与资源压缩 | 已启用 | Release 会执行 R8 和资源压缩，不能用 Debug 行为替代正式版验证。 |
| APK 名称 | `Han1meViewer-v{versionName}.apk` | 由 `androidComponents` 在构建时设置。 |

版本号位于 `app/build.gradle.kts` 的 `defaultConfig`：

```kotlin
versionCode = 260723
versionName = "26.2.1"
```

发布新版本时，`versionCode` 必须大于上一正式版，`versionName` 则使用面向用户的版本名称。应用内更新逻辑会以 `versionCode` 判断版本新旧。

## 2. 创建并保护发布密钥

### 2.1 生成 keystore

在项目根目录打开 PowerShell，执行以下命令。示例将密钥保存到项目根目录的 `keystore/han1meviewer-release.jks`；目录和文件名可自行调整。

```powershell
New-Item -ItemType Directory -Force .\keystore
keytool -genkeypair -v `
  -keystore .\keystore\han1meviewer-release.jks `
  -alias han1meviewer `
  -keyalg RSA `
  -keysize 4096 `
  -validity 10000
```

按提示设置 keystore 密码、密钥密码和证书信息。建议使用密码管理器保存以下内容，并将 `.jks` 文件做离线加密备份：

- keystore 文件；
- keystore 密码；
- alias；
- alias 对应的密钥密码。

这些内容都属于发布凭据，不能提交到 Git，也不要放进公开截图、日志或 Release 附件。

### 2.2 保存本地签名配置

在项目根目录新建 `keystore.properties`，内容如下。路径使用正斜杠，避免 Windows 反斜杠转义问题：

```properties
RELEASE_STORE_FILE=keystore/han1meviewer-release.jks
RELEASE_STORE_PASSWORD=替换为_keystore_密码
RELEASE_KEY_ALIAS=han1meviewer
RELEASE_KEY_PASSWORD=替换为_key_密码
```

将 `keystore.properties` 和 keystore 目录加入本机 `.gitignore`，例如：

```gitignore
/keystore/
/keystore.properties
```

现有项目没有内置 Release 签名配置。请在 `app/build.gradle.kts` 的 `android {` 块之前读取该文件，并在 `android {}` 内配置 `signingConfigs`：

```kotlin
import java.util.Properties

val keystoreProperties = Properties()
val keystorePropertiesFile = rootProject.file("keystore.properties")

if (keystorePropertiesFile.exists()) {
    keystorePropertiesFile.inputStream().use(keystoreProperties::load)
}
```

```kotlin
android {
    // 现有配置保持不变

    signingConfigs {
        create("release") {
            require(keystorePropertiesFile.exists()) {
                "Release 构建需要项目根目录的 keystore.properties"
            }
            storeFile = rootProject.file(
                requireNotNull(keystoreProperties.getProperty("RELEASE_STORE_FILE"))
            )
            storePassword = requireNotNull(
                keystoreProperties.getProperty("RELEASE_STORE_PASSWORD")
            )
            keyAlias = requireNotNull(
                keystoreProperties.getProperty("RELEASE_KEY_ALIAS")
            )
            keyPassword = requireNotNull(
                keystoreProperties.getProperty("RELEASE_KEY_PASSWORD")
            )
        }
    }

    buildTypes {
        release {
            // 保留原有 isMinifyEnabled、isShrinkResources 等配置
            signingConfig = signingConfigs.getByName("release")
        }
    }
}
```

这里故意让缺少 `keystore.properties` 的 Release 构建立即失败，避免误将未签名或使用错误密钥的 APK 当作正式版发布。调试构建不读取这些凭据，仍可正常执行。

## 3. 配置原生签名校验

### 3.1 校验机制说明

原生库 `app/src/main/cpp/chino.cpp` 会从已安装 APK 的 APK Signing Block 中读取签名证书，并计算其 SHA-256 值。该值与 `app/src/main/cpp/chino.h` 中的 `EXPECTED_SIG_HASH` 比较：

```cpp
#define EXPECTED_SIG_HASH "在此填写 64 位小写十六进制 SHA-256"
```

Debug 版本不会启用该拦截；Release 版本进入视频页时会调用 `svc()` 进行校验。校验值绑定的是签名证书，而非 APK 文件内容，因此使用同一发布密钥重新构建后仍然有效。

### 3.2 读取实际签名哈希

项目已经提供了 JNI 方法 `getString()`，其声明在：

```text
app/src/main/java/io/github/daisukikaffuchino/han1meviewer/ui/screen/video/VideoRouteHostScreen.kt
```

在该文件已有的 `LaunchedEffect(Unit)` 中，临时添加一行日志。文件已导入 `android.util.Log`：

```kotlin
LaunchedEffect(Unit) {
    Log.d("SignatureHash", getString())
    // 保留此处原有的 if/else 校验逻辑
}
```

随后先构建一个使用发布密钥签名的 Release APK：

```powershell
.\gradlew.bat :app:assembleRelease
```

产物位于：

```text
app\build\outputs\apk\release\Han1meViewer-v{versionName}.apk
```

使用已安装 Android SDK 的 `adb` 安装这个临时 APK。若设备已安装相同包名但来自不同签名，必须先在设备上卸载旧应用；卸载会清除该应用的数据，请先自行备份。

```powershell
adb install -r .\app\build\outputs\apk\release\Han1meViewer-v26.2.1.apk
```

启动应用并进入任意视频详情页，在 Android Studio Logcat 中按以下条件过滤：

```text
package:io.github.daisukikaffuchino.han1meviewer tag:SignatureHash
```

复制 `getString()` 输出的 64 位小写十六进制字符串。若输出为 `failed`，通常表示 APK 未采用 v2/v3 签名、设备上没有安装当前 APK，或原生库未成功加载；不要回填该值。

也可用 Android SDK Build-Tools 的 `apksigner` 交叉检查证书指纹：

```powershell
apksigner verify --verbose --print-certs .\app\build\outputs\apk\release\Han1meViewer-v26.2.1.apk
```

其中 `Signer #1 certificate SHA-256 digest` 去掉冒号并转换为小写后，应与 Logcat 的值一致。

### 3.3 回填与最终确认

将获得的值替换 `app/src/main/cpp/chino.h` 中的 `EXPECTED_SIG_HASH`，移除前一步临时加入的 `Log.d(...)`，然后重新构建 Release APK：

```powershell
.\gradlew.bat :app:clean :app:assembleRelease
```

重新安装最终 APK，并进入视频详情页。正常情况下不会出现签名校验失败提示。此步骤不能省略：第一次构建仅用于读取证书哈希，第二次构建才是可发布产物。

## 4. 执行构建

### 4.1 命令行构建

在项目根目录执行：

```powershell
# 检查当前 Gradle、JDK 和 Android SDK 解析结果
.\gradlew.bat --version

# 快速编译检查，适合修改 Kotlin/Compose 后使用
.\gradlew.bat :app:compileDebugKotlin

# 运行静态检查
.\gradlew.bat :app:lintRelease

# 构建正式版 APK
.\gradlew.bat :app:assembleRelease
```

`assembleRelease` 会执行资源处理、Kotlin/Java 编译、CMake 原生库编译、R8 混淆、资源压缩、签名和 APK 对齐。首次构建或清理缓存后的构建耗时明显更长。

本项目 Release 配置启用了 ARM64 ABI split，产物为 `arm64-v8a` 设备使用的单一 APK，而不是通用 APK。发布说明中应明确设备要求。

### 4.2 Android Studio 构建

1. 使用 Android Studio 打开项目根目录并等待 Gradle Sync 完成。
2. 确认 Gradle JDK 为 21，并已安装 API 37、NDK 与 CMake 3.22.1。
3. 在 **Build Variants** 中将 `app` 切换为 `release`。
4. 选择 **Build > Generate Signed Bundle / APK**。
5. 选择 **APK**，填写与 `keystore.properties` 相同的 keystore、alias 和密码。
6. 选择 `release`，完成构建。
7. 在输出目录中找到 `Han1meViewer-v{versionName}.apk`。

若 Gradle 中已经按第 2.2 节配置了 `signingConfig`，也可以直接选择 **Build > Build Bundle(s) / APK(s) > Build APK(s)**。无论使用哪种方式，都必须确认最终 APK 的签名证书与 `EXPECTED_SIG_HASH` 对应同一把密钥。

## 5. 发布前验证清单

在上传 Release 前至少完成以下检查：

- [ ] `versionCode` 大于上一正式版，`versionName` 与发布说明一致。
- [ ] 运行 `:app:compileDebugKotlin` 和 `:app:lintRelease` 均成功；如环境允许，再运行 `:app:assembleRelease` 的完整构建。
- [ ] `apksigner verify --verbose --print-certs <apk 路径>` 显示 `Verified using v2 scheme` 或更高版本签名校验通过。
- [ ] 最终 APK 安装在实际 `arm64-v8a` 设备上，能正常启动、进入视频详情页且未触发签名校验失败。
- [ ] 至少检查首页、搜索、登录状态恢复、在线播放、下载入口和设置页等高频路径。
- [ ] 确认 APK 名称、版本号、文件大小与 Release 发布说明一致。
- [ ] 确认未提交或上传 `.jks`、`keystore.properties`、密码、Token、真实 Cookie 和本地日志。

可用以下命令核验 APK 的签名和大小：

```powershell
Get-FileHash .\app\build\outputs\apk\release\Han1meViewer-v26.2.1.apk -Algorithm SHA256
apksigner verify --verbose --print-certs .\app\build\outputs\apk\release\Han1meViewer-v26.2.1.apk
```

`Get-FileHash` 得到的是发布文件校验和，适合在发布页提供给用户核验；它不同于 `EXPECTED_SIG_HASH` 使用的证书哈希，二者不能混用。

## 6. 常见问题

### `SDK location not found` 或找不到 Android API 37

在 Android Studio 的 SDK Manager 安装 Android SDK Platform 37，并确认项目根目录存在仅供本机使用的 `local.properties`，其中 `sdk.dir` 指向 Android SDK 目录。不要提交 `local.properties`。

### `CMake 3.22.1 was not found` 或 NDK 编译失败

在 SDK Tools 中勾选 **CMake 3.22.1** 和 **NDK (Side by side)** 后重新同步。不要只安装系统 CMake；本项目会使用 Android SDK 管理的 CMake 版本构建 `chino` 原生库。

### Release 构建没有签名或签名配置报错

检查 `keystore.properties` 路径、alias、两类密码，以及 `app/build.gradle.kts` 是否已将 `release` 的 `signingConfig` 指向 `signingConfigs.getByName("release")`。先不要尝试用 Debug keystore 绕过错误。

### 安装时提示 `INSTALL_FAILED_UPDATE_INCOMPATIBLE`

设备上已有同包名但签名不同的应用。若不需要保留数据，可卸载旧应用后再安装；若需要保留数据，只能使用旧版本相同的发布密钥重新签名。

### 视频页提示签名校验失败

检查以下项：

- 最终 APK 是否由与临时 APK 相同的发布密钥签名；
- `EXPECTED_SIG_HASH` 是否是 Logcat 中的完整 64 位值，且没有引号、空格或换行错误；
- 回填后是否重新构建并重新安装了 APK；
- 是否误安装了 Debug APK 或其他渠道重新签名的 APK。

### `getString()` 输出 `failed`

确认正在运行的是签名后的 APK，并从已安装应用进入视频详情页后再查看日志。该函数读取当前安装包路径及 APK Signing Block；未签名、仅 v1 签名或安装包路径无法读取时会返回 `failed`。

## 7. 建议的发布顺序

1. 完成功能开发并更新 `versionCode`、`versionName` 和 Release 说明。
2. 用长期保存的发布密钥构建临时 Release APK。
3. 读取并确认签名证书 SHA-256，首次使用该密钥时回填 `EXPECTED_SIG_HASH`。
4. 移除临时日志，重新构建最终 Release APK。
5. 在真机进行安装、签名校验和核心流程回归。
6. 记录最终 APK 的文件 SHA-256，并连同版本说明上传发布。

后续版本只要不更换发布密钥，通常无需再次修改 `EXPECTED_SIG_HASH`；仍建议每次发布都通过 `apksigner --print-certs` 复核签名证书，防止 CI、IDE 或第三方处理流程意外使用其他密钥重新签名。
